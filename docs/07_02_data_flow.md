# 07.02 — Поток данных (Data Flow)

## Общая схема движения данных
```bash
Raw (ClickHouse Bronze)
↓
Staging / bronze          ← Минимальная очистка (только колонки)
↓
Intermediate / prepare     ← Фильтрация + объединение шапки и товаров
↓
Intermediate / enriched / 07_02 /    ← Обогащение из справочников
↓
Marts / data_quality / 07_02 /      ← Расчёт ошибок (отдельные модели по типам документов)
↓
Marts / data_quality / 07_02 / mart_07_02_all_errors   ← Объединение всех ошибок проекта 07_02
↓
Marts / data_quality / mart_all_errors   ← Объединение ошибок всех подпроектов 07_XX
↓
DataLens / BI-дашборды
```

## Детализация по этапам

### Этап 1: Staging (bronze)
- Назначение: Минимальная очистка от ненужных столбцов.
- Материализация: `view`
- Что делается: Отбор нужных колонок, приведение типов (минимально).
- Примеры моделей:
  - `stg_bronze__doc_check_kkm_header.sql`
  - `stg_bronze__doc_check_kkm_tab_goods.sql`
  - `stg_bronze__dim_organization.sql`
  - ... (полный список см. в `07_02_architecture.md`)

### Этап 2: Prepare
- Назначение: Объединение (при необходимости) документа и номенклатуры из этого документа (денормализованное представление).
- Материализация: `table` (запланирован рефакторинг на `incremental` с отслеживанием изменений по `exported_at` / `partition_day`)
- Что делается: Объединение шапки и табличной части с номенклатурой (`FULL JOIN`)
- Примеры моделей:
  - `int_check_kkm_prepared.sql`
  - `int_refund_sale_doc_prepared.sql`
  - ... (полный список см. в `07_02_architecture.md`)

### Этап 3: Enriched
- Назначение: Обогатить факты данными из справочников.
- Материализация: `view`
- Что делается: `LEFT JOIN` со справочниками организаций, сотрудников, номенклатуры.
- Примеры моделей:
  - `int_07_02_check_kkm_enriched.sql`
  - `int_07_02_sale_doc_enriched.sql`
  - ... (полный список см. в `07_02_architecture.md`)

### Этап 4: Data Quality (отдельные mart'ы с errors)
- Назначение: Выявить записи (факты) с ошибками в ставках НДС.
- Материализация: `table`
- Что делается: Применение бизнес-правил ошибок (количество категорий зависит от типа документа). Описания ошибок приведены в `07_02_business_description.md`
- Примеры моделей:
  - `mart_07_02_check_kkm_errors.sql`
  - `mart_07_02_refund_sale_doc_errors.sql`
  - ... (полный список см. в `07_02_architecture.md`)

### Этап 5: Витрина ошибок, общая для подпроекта 07_02
- Назначение: Собрать все ошибки проекта 07.02 в одну таблицу.
- Материализация: `view`
- Модель: `mart_07_02_all_errors.sql`
- Содержит: `UNION ALL` всех моделей из папки `07_02/`

### Этап 6: Финальная витрина ошибок
- Назначение: Собрать все ошибки проекта Data Quality в одну таблицу.
- Материализация: `view`
- Модель: `mart_all_errors.sql`
- Содержит: `UNION ALL` всех `mart_*_all_errors.sql`

## Принципы потоков данных

- Данные движутся **сверху вниз** по слоям.
- Каждый этап имеет чёткую ответственность.
- Обогащение (`enriched`) сделано view, чтобы не хранить данные повторно.
- Расчёт ошибок вынесен в отдельные модели для независимого пересчёта.
- Финальная витрина — лёгкая и не хранит данные.
