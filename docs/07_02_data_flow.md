# 07.02 — Поток данных (Data Flow)

## Общая схема движения данных
Raw (ClickHouse Bronze)
↓
Staging / bronze          ← Минимальная очистка (только колонки)
↓
Intermediate / 07_02 / prepare     ← Фильтрация + объединение шапки и товаров
↓
Intermediate / 07_02 / enriched    ← Обогащение справочниками
↓
Marts / data_quality / 07_02 / errors     ← Расчёт ошибок (отдельные модели)
↓
Marts / data_quality / 07_02 / mart_07_02_all_errors   ← Объединение всех ошибок
↓
DataLens / BI-дашборды
text## Детализация по этапам

### Этап 1: Staging (bronze)
- Назначение: Привести сырые данные к единому виду.
- Материализация: `view`
- Что делается: Отбор нужных колонок, приведение типов (минимально).
- Примеры моделей:
  - `stg_bronze__doc_check_kkm_header.sql`
  - `stg_bronze__doc_check_kkm_goods.sql`
  - `stg_bronze__dim_organization.sql`

### Этап 2: Prepare
- Назначение: Подготовить данные под конкретный тип документа.
- Материализация: `table` (рекомендуется incremental)
- Что делается:
  - Фильтрация записей (по дате, статусу, проведённости и т.д.)
  - Объединение шапки и табличной части (`LEFT JOIN`)
- Примеры моделей:
  - `int_07_02_chek_kkm_prepared.sql`
  - `int_07_02_refund_sale_doc_prepared.sql`

### Этап 3: Enriched
- Назначение: Обогатить данные справочниками.
- Материализация: `view`
- Что делается: `LEFT JOIN` с организациями, сотрудниками, номенклатурой.
- Примеры моделей:
  - `int_07_02_chek_kkm_enriched.sql`
  - `int_07_02_sale_doc_enriched.sql`

### Этап 4: Errors (отдельные mart'ы)
- Назначение: Выявить записи с ошибками ставок НДС.
- Материализация: `table`
- Что делается: Применение бизнес-правил ошибок (3 категории).
- Примеры моделей:
  - `mart_07_02_chek_kkm_errors.sql`
  - `mart_07_02_refund_sale_doc_errors.sql`

### Этап 5: Общая витрина ошибок
- Назначение: Собрать все ошибки проекта 07.02 в одну таблицу.
- Материализация: `view`
- Модель: `mart_07_02_all_errors.sql`
- Содержит: `UNION ALL` всех моделей из папки `errors/`

## Принципы потоков данных

- Данные движутся **сверху вниз** по слоям.
- Каждый этап имеет чёткую ответственность.
- Обогащение (`enriched`) сделано view, чтобы не хранить данные повторно.
- Расчёт ошибок вынесен в отдельные модели для независимого пересчёта.
- Финальная витрина — лёгкая и не хранит данные.
