# 07.02 — Ошибки ставок НДС (Data Quality)

## 1. Цель подпроекта
Выявление в системе 1С документов, подлежащих передаче в ОФД/ФНС (продаж, оплат и возвратов от клиента) и содержащих некорректные, на момент проверки, ставки НДС.

## 2. Архитектура преобразований (5 этапов)

| Этап | Название слоя                                    | Тип материализации    | Описание                                                                  |
|------|--------------------------------------------------|-----------------------|---------------------------------------------------------------------------|
| 1    | `staging/bronze`                                 | `view`                | Минимальная очистка сырых данных (только отбор колонок)                   |
| 2    | `intermediate/prepare`                           | `table` + incremental | Объединение шапки и товаров под конкретный тип документа (общий слой)     |
| 3    | `intermediate/enriched/07_02`                    | `view`                | Обогащение из справочников (по подпроектам)                               |
| 4    | `marts/data_quality/07_02/`                      | `table`               | Расчёт ошибок по каждому типу документа (отдельные mart'ы)                |
| 5    | `marts/data_quality/07_02/mart_07_02_all_errors` | `view`                | Объединение всех ошибок проекта 07.02                                     |

## 4. Правила именования

- `stg_bronze__<тип объекта>__<header|tab_goods>`
- `int_<документ>_prepared`
- `int_07_02_<документ>_enriched`
- `mart_07_02_<документ>_errors`
- `mart_07_02_all_errors`

- Планируется: в следующей итерации после отправки в GIT перейти на единый стандарт именования dbt с использованием double underscore (__) для лучшей читаемости (например, `int_check_kkm__prepared`).

## 5. Принципы архитектуры

- Максимальная изоляция по типам документов на уровне `prepared`, с фиксацией в table для возможности переиспользования без пересчёта.
- Слой `prepared` является общим для всех подпроектов.
- Слой обогащения `enriched` материализован как `view`, потому что:
    - Справочники (особенно «Номенклатура») меняются ежедневно, преимущественно за счёт добавления новых элементов.
    - Поле, которое нас интересует (ставка НДС), у существующих записей меняется крайне редко.
    - Нет смысла хранить и отслеживать изменения в обогащённых данных — достаточно получить текущую версию, при необходимости рассчитать ошибки.
- Расчёт ошибок вынесен в отдельные mart-модели (можно пересчитывать независимо).
- Финальная витрина — лёгкая view.

## 6. Ограничения по данным (из описания проекта 07_02)

- Дата документа ≥ 2026-01-01 (кроме возвратов)
- Только проведённые документы.
- Чек ККМ — только статус "Чек пробит"
- Номенклатура — используются только льготные ставки НДС (10)
- Все ставки НДС - используются только основные ставки (5, 7, 10, 22). Авансовые ставки (10/110) в документах 1С - не используются.
